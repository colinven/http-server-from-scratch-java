# my-http-server

A minimalistic HTTP/1.1 server in Java, built from scratch to learn how web servers actually work — request parsing, routing, response serialization, and concurrency, with no framework underneath.

## Why this exists

This is a learning project. The goal was not to ship another web framework, but to sit with the HTTP/1.1 spec long enough to implement it end-to-end: byte-level parsing off a raw `InputStream`, a routing table, a response serializer, and a thread pool that survives slow handlers, exceptions, and shutdown. It is intentionally minimal — useful as a study, not as a production server.

## Features

- **HTTP/1.1 request parsing** via a hand-written state machine (`RequestParser` → `REQUEST_LINE → HEADER_NAME → HEADER_VALUE → BODY`), enforcing CRLF line endings, mandatory `Host` header, and the HTTP/1.1 version.
- **Routing** for `GET` / `POST` / `PUT` / `DELETE` with exact-path matching (`Router`).
- **Query string parsing** via `HttpRequest.queryParams()`.
- **Request bodies** read using `Content-Length`.
- **Fixed thread pool** (50 workers) for concurrent connections, with the main thread blocking on `accept()`.
- **Graceful shutdown** via a JVM shutdown hook that drains in-flight work for up to 30 seconds.
- **Full status code coverage** (100–511) with reason phrases auto-generated by `HttpResponse`.
- **Structured error handling**: parse errors map to `400` / `505`, uncaught handler exceptions to `500`, and the server keeps running either way.

## Non-goals

Things this server deliberately does **not** do — calling them out so the scope is honest:

- No HTTPS / TLS
- No route parameters (e.g. `/users/:id`) — exact path matching only
- Only `text/plain` response helpers; other content types must be assembled by hand
- HTTP/1.1 only — HTTP/1.0 and HTTP/2 are rejected
- No keep-alive or connection reuse — one request per connection
- No middleware / filter chain
- No request body streaming — the full body is buffered in memory

## Architecture

The lifecycle of a single request:

```
                                    ┌──────────────────────────────────────┐
   client ──TCP──> ServerSocket ──> │ pool worker (1 of 50)                │
                   (main thread)    │                                      │
                                    │   RequestParser   → HttpRequest      │
                                    │   Router          → RouteHandler     │
                                    │   RouteHandler    → HttpResponse     │
                                    │  HttpResponse.write() ──TCP──> client│
                                    └──────────────────────────────────────┘
```

1. `HttpServer` accepts a socket on the main thread and submits it to a worker in the fixed thread pool.
2. `HttpConnectionHandler` runs on the worker:
   - `RequestParser` reads bytes through the state machine and produces an `HttpRequest`.
   - `Router` looks up a `RouteHandler` by `(HttpMethod, path)`; missing routes return `404`.
   - The handler returns an `HttpResponse`.
   - `HttpResponse.write()` serializes the status line, headers, and body back over the socket.
3. `HttpParseException` is caught and rendered as `400` (bad request) or `505` (unsupported HTTP version). Anything else uncaught becomes `500`. The worker returns to the pool; the server keeps accepting.

Key files:
- `src/main/java/com/myhttpserver/app/server/HttpServer.java` — accept loop + thread pool + shutdown hook
- `src/main/java/com/myhttpserver/app/io/HttpConnectionHandler.java` — per-connection pipeline
- `src/main/java/com/myhttpserver/app/parser/RequestParser.java` — state-machine parser
- `src/main/java/com/myhttpserver/app/router/Router.java` — `(method, path)` → handler
- `src/main/java/com/myhttpserver/app/response/HttpResponse.java` — response model + serialization
- `src/main/java/com/myhttpserver/app/config/AppConfig.java` — route registrations

## Build & run

Requires Java 17 and Maven.

```bash
mvn clean package
java -jar target/http-server.jar
```

The server listens on **port 8080**. Routes are registered in `AppConfig.java` — add your own there.

## Example requests

With the server running, the routes wired in `AppConfig` produce:

```bash
# default greeting
curl http://localhost:8080/hello
# → Hello, World!

# query parameter
curl 'http://localhost:8080/hello?name=Colin'
# → Hello, Colin!

# second route
curl http://localhost:8080/hello/goodbye
# → Goodbye, World!

# slow handler — sleeps 2s. Run this in one shell and /hello in another to
# confirm the slow request does NOT block fast ones.
curl http://localhost:8080/slow
# → Slow response

# uncaught exception in a handler — server returns 500 and keeps running
curl -i http://localhost:8080/explode
# → HTTP/1.1 500 Internal Server Error

# unknown route
curl -i http://localhost:8080/does-not-exist
# → HTTP/1.1 404 Not Found
```

## Tests

```bash
mvn test
```

The suite covers two layers:

- **Parser unit tests** (`RequestParserTest`) — valid requests, header parsing, whitespace handling, and the rejection paths: bare LF, missing `Host`, HTTP/1.0, malformed request line.
- **Server integration tests** (`HttpServerIntegrationTest`) — real sockets, real thread pool. Asserts that responses don't interleave under concurrent clients, that slow handlers don't starve fast ones, that the pool doesn't leak threads under 1000 requests, that graceful shutdown drains in-flight work, and that a handler exception yields `500` without taking the server down.

## What I learned

- **HTTP/1.1 framing is genuinely small but surprisingly strict on the wire.** CRLF (not LF) terminates lines, the `Host` header is mandatory, and the body length is whatever `Content-Length` says it is — there is no other framing signal in a vanilla request.
- **A state machine is the right shape for a streaming parser.** Trying to "read a line, then parse it" tangles the byte loop with the grammar; a state-per-grammar-position keeps both clean and makes error positions obvious.
- **Thread-per-connection is the simplest concurrency model that works** — and it scales until it doesn't. A fixed pool caps damage from misbehaving clients, but the moment you want tens of thousands of idle connections, NIO (or virtual threads) becomes the right answer.
- **`Content-Length` is load-bearing.** Without it the parser has no way to know where the body ends; with it, the body read is a fixed-size byte read, not a "read until something."
- **Testing concurrent server code without flakiness means using the real thing.** Mocked sockets prove the parser compiles; real sockets, a real thread pool, and assertions on observable invariants (no interleaving, no leaks, shutdown drains) prove the server actually works.

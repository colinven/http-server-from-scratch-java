Building an HTTP Server & Understanding the Web

A structured, max-depth curriculum for constructing a production-quality HTTP/1.1 server from first principles — vanilla Java, no frameworks, RFC in hand.

Cadence
~10 hrs / wk
Duration
8 weeks
Modules

The goal of this curriculum is not to build an HTTP server. The goal is to understand what HTTP actually is, to see through the abstractions that Tomcat and Spring Boot have quietly wrapped around you, and to emerge with the kind of foundational fluency that changes how you read every backend codebase afterward.

You will read RFCs. You will write parsers that reject malformed bytes. You will be attacked by Slowloris and survive. You will rewrite your I/O layer three times — thread-per-connection, NIO selector loop, and virtual threads — and feel in your hands why each one exists. And at the end, you will have a server you wrote, tested, hardened, and benchmarked.

Each module below follows the same shape: core topics, resources, learning outcomes, and where code is produced, test cases written as JUnit + scripted integration tests. Diagrams come before code. Concepts come before keystrokes.

Module 00 · Week 1 · First Half ~5 hours

Foundations

Before a single line of server code — what a socket is, what TCP does for you, and the exact bytes that constitute an HTTP request.

Core Topics
  • TCP/IP The three-way handshake, ordered byte streams, connection lifecycle (SYN, ACK, FIN), and why TCP gives you "a reliable pipe" rather than discrete messages.
  • Sockets What an OS socket actually is — a file descriptor bound to an (IP, port) pair. The server/client asymmetry. listen(), accept(), connect().
  • HTTP/1.1 shape Request line, headers, blank line, body. CRLF (\r\n) as the protocol's delimiter. Why \r\n\r\n ends the header section.
  • Status codes 1xx–5xx classes, the ones you'll actually return (200, 301, 400, 404, 500), and the ones you'll learn to hate (408, 413, 431).
  • Mental model A server is a loop: accept → read bytes → parse → respond → close (or keep-alive).

Module 01 · Week 1 · Second Half ~5 hours · First code

The Echo Server

Your first working TCP program in Java. No HTTP yet — just bytes in, bytes out.

Core Topics
  • ServerSocket new ServerSocket(8080), the accept() call that blocks until a client connects, and why that blocking matters.
  • Streams Socket.getInputStream() and getOutputStream(). Why you wrap them in BufferedReader/BufferedWriter (spoiler: syscall overhead).
  • Resource management try-with-resources on sockets. What happens if you forget to close.
  • Testing network code How to write an integration test that spins up a real server on an ephemeral port, sends bytes to it, asserts the response.
What You'll Build

A server that accepts one connection at a time, reads a line of text, echoes it back uppercased, and closes. That's it. No HTTP. The point is to feel sockets in your hands before adding protocol logic on top.

public class EchoServer {
  public static void main(String[] args) throws IOException {
    try (ServerSocket server = new ServerSocket(8080)) {
      System.out.println("listening on 8080...");
      while (true) {
        try (Socket client = server.accept()) {   // blocks
          // read one line, uppercase it, write it back
        }
      }
    }
  }
}

Module 02 · Week 2 ~10 hours

Request Parsing

From bytes to structured data. This is where you start being an HTTP server and not a TCP echo program.

Core Topics
  • Request line METHOD SP URI SP HTTP-VERSION CRLF. The three fields, the exact whitespace rules, why case matters for the method.
  • Header parsing field-name: field-value, case-insensitive names, folding rules (obsolete but present in the wild), multi-value semantics.
  • Reading until blank line The \r\n\r\n header terminator. Why you must read exactly until that sequence and not one byte further.
  • Domain modeling Java records for HttpRequest, RequestLine, Headers. Why immutable value types are the right abstraction here.
  • Error responses Return 400 Bad Request on malformed input. Return 505 HTTP Version Not Supported for non-1.1. These responses must themselves be well-formed HTTP.

Design note Do not use BufferedReader.readLine() in production code for parsing HTTP. It silently handles \n, \r, and \r\n identically — which is exactly the kind of parser leniency that leads to request smuggling vulnerabilities. Read bytes directly. Require \r\n. Reject bare \n. You can use it for a first-pass prototype, but note this and return.

Module 03 · Week 3 ~10 hours

Concurrency

One client at a time is a toy. Many clients at once is a server. Thread-per-connection, then a bounded pool.

Core Topics
  • Thread-per-connection Spawn a new Thread for each accepted socket. Why this is the simplest model and why it breaks at ~10K clients.
  • Bounded thread pools Executors.newFixedThreadPool(n). Queueing semantics. What happens when the pool saturates.
  • Graceful shutdown Runtime.addShutdownHook, ExecutorService.shutdown() vs shutdownNow(), draining in-flight requests.
  • Shared state Every handler runs in its own thread. Mutable state on the server object requires synchronization. Prefer immutability.
  • The C10K problem The historical context. Why this is the motivation for NIO (Module 07) and virtual threads (Module 08).
Architecture decision Build this with an abstraction boundary over the I/O layer — something like a ConnectionHandler interface whose implementation is swappable. You will have three implementations before the end: thread-per-connection here, NIO selector in Module 07, virtual thread per task in Module 08. Designing for pluggability now makes the later rewrites teach you something instead of making you start over.
Resources
ARTICLE Dan Kegel — "The C10K Problem" The canonical write-up. Read the first half; the specific APIs discussed are dated but the framing is timeless. You will cite this in interviews.
BOOK Java Concurrency in Practice — Brian Goetz, chapters 6 & 8 Chapter 6 on Executors, Chapter 8 on pool tuning. Read once; reference forever.
ARTICLE "Understanding the Java Executor Service" Any well-regarded walkthrough. Focus on: what happens when the queue fills, what RejectedExecutionException means.

Module 04 · Week 4 ~10 hours

Routing, Bodies, and Static Files

Your server starts to look like something you could actually ship.

Core Topics
  • Routing Map (method, path) to handler functions. Support exact-match and simple path parameters like /users/{id}.
  • Handlers as interfaces Handler = Function<HttpRequest, HttpResponse>. This is what Spring is hiding under @GetMapping. Now you know.
  • Request bodies Read exactly Content-Length bytes from the stream. Not one byte less, not one more.
  • Static file serving Map URL paths to files under a root directory. Correctly set Content-Type by extension.
  • Directory traversal defense GET /../etc/passwd must not escape the root. Canonicalize paths and verify containment.
  • MIME types A small lookup table. HTML, CSS, JS, JSON, plain text, the common image types.
Security checkpoint Directory traversal is the first vulnerability you must defend against. Before you ship any static file handler, prove via test that GET /../../../etc/passwd, GET /..%2f..%2fetc/passwd, and GET /static/../../private.txt all return 403 or 404 — never the target file. Canonicalize via Path.toRealPath(), then verify startsWith(root).

Module 05 · Week 5 ~10 hours

Keep-Alive & Chunked Encoding

The two features that make HTTP/1.1 fast, and the two features most often implemented incorrectly.

Core Topics
  • Persistent connections HTTP/1.1 default is keep-alive. One TCP connection, many requests. Massive perf win over connect-per-request.
  • When to close Connection: close header, idle timeout, malformed request, server shutdown. Each has a specific handling.
  • Framing in a persistent stream You cannot rely on connection close to mark the end of a response. You must use Content-Length or Transfer-Encoding: chunked.
  • Chunked transfer encoding The format: hex size + CRLF + data + CRLF, ending with 0\r\n\r\n. Required for responses whose length you don't know up front.
  • Trailers Headers at the end of a chunked body. Rarely used, but in the spec.
  • Idempotency and retry semantics Why GET can be retried on a persistent connection, why POST generally cannot.
Why this module matters Keep-alive and chunked encoding together are where HTTP gets subtle. They're also where an enormous class of vulnerabilities lives — request smuggling (Module 06) fundamentally depends on parsers disagreeing about where one request ends and the next begins. You cannot defend against something you haven't implemented, which is why this module comes before hardening.
Resources
RFC RFC 9112 §6 (Message Body Length), §7 (Transfer Codings), §9 (Connection Management) These three sections together are the entire keep-alive + chunked contract. Read them as one unit.
ARTICLE "HTTP/1.1 persistent connections explained" Any quality deep-dive. MDN's page on connection management is a good start.

Module 06 · Week 6 ~10 hours · Adversarial

Hardening

Your server works. Now you break it on purpose, then fix it. This is where most tutorials stop and where the real learning begins.

Core Topics
  • Slowloris Many connections, each sending headers one byte per 10 seconds. Your thread pool drowns. Fix: header read timeout.
  • Slow POST Same attack against the request body. Fix: minimum data rate, total body-read timeout.
  • Oversize attacks 1GB request line, 1GB header, 10GB body. Fix: hard limits on line length (8KB), header count (100), header size, body size.
  • Header injection / CRLF injection Never copy untrusted input into response headers without rejecting bare CR/LF bytes.
  • Request smuggling CL.TE and TE.CL desync attacks. Your defense from Module 05 (reject both present) handles the common case. Module 06 adds the nuances.
  • Connection limits Maximum concurrent connections total. Optional: per-IP caps.
  • Resource exhaustion File descriptor limits. Memory bounds on buffered reads.
The Adversarial Drill

For each attack below, you write a test that exploits it against your current server, watch it succeed (server crashes, hangs, or leaks resources), implement the defense, and the same test now asserts the defense works. This is the single highest-value exercise in the whole curriculum.

Module 07 · Week 7 ~10 hours · Rewrite

NIO Rewrite

Swap out the I/O layer for a non-blocking selector loop. You'll feel, viscerally, why Netty exists.

Core Topics
  • The motivation Revisit C10K. One thread per connection has an OS-level cost (stack, scheduler entry, context switch). NIO lets one thread monitor thousands of connections.
  • Non-blocking mode SocketChannel.configureBlocking(false). Reads return immediately, possibly with zero bytes. You loop.
  • Selector Register channels with a Selector. select() blocks until any registered channel is ready (readable, writable, or has a new connection).
  • Readiness, not data A key is "readable" when read() would not block, but you still might get 0 bytes (EOF). Get this wrong and you infinite loop.
  • Attachments and state machines Each channel needs a per-connection state object. Because read can return partial data, your parser must be incremental — resumable between select() calls.
  • The event loop model This is Node.js, Netty, nginx, libuv — the shape of every high-performance server from ~2005 to the advent of Loom.
Why the rewrite is the point You're not rewriting because NIO is better — you're rewriting because implementing the same protocol in two concurrency models teaches you what the protocol was, independent of its implementation. The request parser from Module 02 probably can't be reused as-is here; it assumed it could block on the input stream. You'll adapt it into a state machine. That adaptation is the lesson.

Module 08 · Week 8 ~10 hours · The payoff

Virtual Threads

You have two working implementations. Now a third. Loom collapses the historical dilemma — and you'll see exactly how.

Core Topics
  • What virtual threads are JDK 21+ lightweight threads. Scheduled by the JVM onto a small pool of carrier (platform) threads. Blocking calls park the virtual thread without parking the carrier.
  • Why they change the game The old dilemma ("simple blocking code OR scalable event loop, pick one") dissolves. Blocking code scales again.
  • Implementation swap Executors.newVirtualThreadPerTaskExecutor(). One virtual thread per accepted connection. That's it. The thread-per-connection model from Module 03 becomes the production-grade model.
  • Pinning The main footgun. synchronized blocks and JNI calls pin the virtual thread to its carrier, defeating the scalability. Replace with ReentrantLock.
  • Benchmarking Compare the three implementations: thread-per-connection (platform), NIO selector, virtual-thread-per-connection. Under what workloads does each win?
  • Reflection You have three working servers. Write a README section comparing them. This is the capstone artifact.
Structural observation Your Module 03 code is almost correct for virtual threads already. That's the profound insight: with Loom, the simplest concurrency model (one thread per connection) becomes the scalable one. Two decades of NIO complexity exists to solve a problem that Loom largely dissolves. Understanding this both historically and mechanically is the payoff of this curriculum.
